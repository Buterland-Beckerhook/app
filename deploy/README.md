# Deployment (Beta & Prod)

The container is built by **GitHub Actions** and pushed to **GHCR**
(`ghcr.io/buterland-beckerhook/app`). The server **does not build** — it only pulls
the finished image. The image is **multi-arch (`linux/amd64` + `linux/arm64`)**; the
prod server is arm64/v8.

- Build workflow: `.github/workflows/build.yml` — runs on push to **`main`** and via
  "Run workflow" (workflow_dispatch). Tags: `:beta` (rolling) + `:sha-XXXXXXX`
  (immutable, for rollback). **Beta tracks `main`; Prod pins an immutable `:sha`.**
- A single `docker compose` stack (`deploy/compose.yml`) serves **Beta and Prod** — the
  difference lives entirely in `.env` (`IMAGE`, `PHX_HOST`, `TRAEFIK_NAME`, `TRAEFIK_RULE`,
  `TRAEFIK_MIDDLEWARES`, `BASIC_AUTH_USERS`), all generated by `setup.sh`.

> Building before the rewrite has merged: the push trigger only fires on `main`, so
> until then build ad-hoc via **GitHub → Actions → Build & push image → Run workflow**
> against your branch.

## Reverse proxy: central Traefik

The stack ships **no reverse proxy of its own** and publishes **no ports**. TLS
(Let's Encrypt), the HTTP→HTTPS redirect, compression, the `www`→apex redirect (Prod)
as well as Basic-Auth + `noindex` (Beta) are all handled by the **central Traefik** on
the host, driven by Docker labels on the `phoenix` service (see `compose.yml`).

Prerequisites: the external Traefik network **`proxy`** already exists on the host, and
Traefik knows the referenced file-provider building blocks `https-redirect@file`,
`secure-tls@file` and the certresolver `le`. (Both are already the case in the existing
setup.)

## First-time setup on the server

1. **DNS:** A record (`beta.buterland-beckerhook.de` for Beta) → server IP. Ports
   80/443 are already served by the central Traefik — this stack opens none itself.

2. **Get the `deploy/` folder onto the server** (git clone the repo or copy the folder).

3. **Generate `.env` with the setup script** (from inside `deploy/`):
   ```sh
   ./setup.sh          # interactive; or: ./setup.sh --yes   (accept generated defaults)
   #                     ./setup.sh --prod  presets prod-mode defaults (default is beta)
   ```
   The script is **idempotent** — re-running only fills values that are still missing;
   existing values are preserved. It:
   - generates every secret **locally with `openssl`** — `SECRET_KEY_BASE`, the
     **VAPID** keypair (Web Push), `ALTCHA_HMAC_KEY`, and a URL-safe `DB_PASSWORD`.
     No image or Mix is needed, so there is **no ordering dependency** on the GHCR pull.
   - for Beta, offers **Basic-Auth**: enter a username (or accept `beta`) and a password
     (or let it generate one — printed **once**, save it). It hashes with bcrypt and
     writes the value **`$`-escaped** so Compose interpolation keeps the hash intact.
   - sets `TRAEFIK_*` from the chosen mode/host and writes `.env` with `chmod 600`,
     then self-checks it with `docker compose config`.
   - You are prompted for the rest (hostname, SMTP, Matomo…) with sensible defaults.

   > **Do not hand-edit secrets or the auth hash.** If you must edit `.env` by hand,
   > remember every literal `$` has to be doubled to `$$` (Compose interpolates `.env`
   > values). This is exactly why Basic-Auth failed silently before. Re-run `setup.sh`
   > rather than editing.

4. **Access the image on GHCR** (the package is private by default) — either log in once
   with a fine-grained PAT that has `read:packages`:
   ```sh
   echo <PAT> | docker login ghcr.io -u <github-user> --password-stdin
   ```
   or set the GHCR package to **public** in its settings (the image holds no secrets —
   those come at runtime from `.env`), which removes the login step. Make sure the
   `:beta` image exists (push to `main`, or run the workflow via workflow_dispatch).

5. **(Optional) seed data — see [Restoring a dump](#restoring-a-dump).** Drop a backup
   into `deploy/restore/` before the first `up` to launch with real content.

6. **Start the stack:**
   ```sh
   docker compose --env-file .env up -d
   ```
   Postgres comes from the public image, Phoenix from GHCR. On boot: the one-shot
   `restore` step runs (a no-op unless `restore/` holds a dump), then Phoenix runs
   `bin/migrate` and starts the server. Traefik picks up the new container from its labels.

### Verify success

```sh
docker compose --env-file .env ps                             # all services "healthy"
curl -fsS https://beta.buterland-beckerhook.de/health/liveness # 200 (once TLS is issued)
```
In the browser: the Basic-Auth dialog (Beta), then the home page renders; the response
carries `X-Robots-Tag: noindex, nofollow` (Beta). Admin login at `/users/log-in`.

## Restoring a dump

The stack can restore a backup automatically on boot. Anything dropped into
`deploy/restore/` is picked up by the one-shot `restore` service **before** Phoenix
starts (Phoenix then migrates on top, so a slightly older dump is still brought
up-to-date):

**Initial seed** (fresh stack, before the first `up`):
```sh
cp bbh-2026-07-16.dump        deploy/restore/     # DB (pg_dump -Fc, from backup.sh)
cp uploads-2026-07-16.tar.gz  deploy/restore/     # optional: uploaded originals
docker compose --env-file .env up -d
```

**Re-seed a running deployment** — ⚠️ stop Phoenix first. The restore `--clean`s the
DB, which would fight (or corrupt) a live app still holding connections:
```sh
docker compose --env-file .env stop phoenix
cp bbh-2026-07-16.dump deploy/restore/
docker compose --env-file .env up -d            # restore runs, then Phoenix restarts
```

- Restore runs **only when a `*.dump` is present**; on success **every** dump/uploads
  file is renamed to `*.restored`, so restarts never re-restore. Drop a fresh `*.dump`
  in to restore again. Keep to **one dump at a time** — the newest is used.
- ⚠️ **Destructive:** restoring `--clean`s and overwrites the current database.
  Intended for (re)seeding — not an "apply diff".
- A failed restore keeps Phoenix down (rather than serving a half-restored DB); check
  `docker compose --env-file .env logs restore`.

## Update flow ("practice")

1. Merge the change to `main` → CI builds a fresh `:beta` image.
2. On the server, pull the new image and restart the container:
   ```sh
   docker compose --env-file .env pull phoenix
   docker compose --env-file .env up -d phoenix
   ```
   Migrations run automatically on restart. Short downtime is fine for Beta.

### Rollback

Every build is additionally tagged as an immutable `:sha-XXXXXXX` (short commit SHA, see
the Actions log or GHCR). To roll back to an earlier state:
```sh
# in .env:  IMAGE=ghcr.io/buterland-beckerhook/app:sha-abc1234
docker compose --env-file .env pull phoenix
docker compose --env-file .env up -d phoenix
```

## Prod

Run `./setup.sh --prod` for its own `.env`. Prod defaults differ from Beta:
- `PHX_HOST=buterland-beckerhook.de`, `TRAEFIK_NAME=bb`,
  `TRAEFIK_RULE=Host(\`buterland-beckerhook.de\`, \`www.buterland-beckerhook.de\`)`,
  `TRAEFIK_MIDDLEWARES=bb-compress,bb-www` (compression + www→apex redirect, **without**
  Basic-Auth/noindex).
- Pin `IMAGE` to a stable `:sha-XXXXXXX` tag (setup.sh prompts for it).

Beta and Prod run on the same Traefik — so they use **different `TRAEFIK_NAME`s**
(`bb-beta` vs. `bb`), otherwise router/middleware names collide.

## Backups

`deploy/backup.sh` — nightly `pg_dump` + uploads, offsite via Borg (cron as root). To
restore one of its archives, copy the `.dump` (and `uploads-*.tar.gz`) into
`deploy/restore/` and run `docker compose up -d` — see [Restoring a dump](#restoring-a-dump).
